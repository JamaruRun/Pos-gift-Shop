# PRD — ระบบ POS + จัดการสต็อก ร้านโชห่วย (Gift Store)

> เอกสารกำหนดความต้องการผลิตภัณฑ์ (Product Requirements Document)
> เวอร์ชัน: 1.0 (MVP) · อัปเดต: 2026-06-21 · สถานะ: ยืนยันแล้ว

---

## 1. ภาพรวมโปรเจกต์ (Project Overview)

ระบบขายหน้าร้าน (POS) พร้อมจัดการสต็อกบนเว็บ สำหรับร้านขายของชำ/ร้านสะดวกซื้อขนาดเล็กในไทย
สินค้า: ขนม, เครื่องดื่ม, เบียร์, บุหรี่, ของใช้ในบ้าน

เป้าหมายหลัก: **เจ้าของร้าน (Gift) ออกจากร้านได้โดยยังคุมยอดขาย เงินสด สต็อก และพฤติกรรมพนักงานได้จากระยะไกล**

คติการออกแบบ: **"ไม่ซับซ้อน ดูง่าย"** — พนักงานใหม่เข้าใจการใช้งานพื้นฐานภายในไม่กี่นาที

---

## 2. ปัญหาธุรกิจ (Business Problem)

| ปัญหา | ผลกระทบ |
|---|---|
| เงินสดรั่ว (ขายไม่ลงระบบ เก็บเงินเอง) | ช่องโหว่ทุจริตอันดับ 1 |
| ของหายโดยไม่รู้สาเหตุ | แยกไม่ออกว่าขาย/ขโมย/ของเสีย/คีย์ผิด |
| ไม่รู้กำไรจริง | ขายดีแต่เงินไม่เหลือ |
| ของขาด/ของล้น | ของขายดีหมดไม่รู้ตัว, ของจมทุน |
| คุมร้านจากระยะไกลไม่ได้ | ต้องเฝ้าหน้าร้านเอง |

---

## 3. กลุ่มผู้ใช้ (Target Users)

- **เจ้าของร้าน (Gift)** — ไม่ใช่สายเทคนิค ต้องการความเรียบง่าย เชื่อถือได้ เร็ว และมองเห็นทุกอย่าง
- **พนักงานหน้าร้าน** — เปลี่ยนคนบ่อย ต้องเรียนรู้เร็ว เน้นขายของให้เร็วที่สุด

---

## 4. บทบาทผู้ใช้ (User Roles)

### 4.1 เจ้าของ (Owner) — สิทธิ์เต็ม
- ดูรายงาน, กำไร, ต้นทุน
- จัดการสินค้า / พนักงาน
- ดู activity log
- ตั้งค่าระบบ (รวมตั้งค่า LINE)
- void บิล (พร้อมเหตุผล)

### 4.2 พนักงาน (Employee) — สิทธิ์จำกัด
- ขายสินค้า
- ค้นหาสินค้าด้วยชื่อ
- เช็คสต็อก (เห็นจำนวน ไม่เห็นต้นทุน)

**พนักงานทำไม่ได้ (โดยค่าเริ่มต้น):** รับของเข้า, แก้ต้นทุนสินค้า, เปิด/ปิด cash session, ดูกำไร/ต้นทุน, ดูรายงาน sensitive, ลบประวัติการขาย, ลบ/แก้ activity log, เข้าตั้งค่าระบบ, จัดการพนักงาน

> **เหตุผลของการจำกัดสิทธิ์:** เป้าหมายธุรกิจหลักคือ **ลดการทุจริตของพนักงานและการปั่นสต็อก** จึงตัดสิทธิ์ที่กระทบเงิน/ต้นทุน/สต็อกออกจากพนักงานโดยค่าเริ่มต้น
>
> **Feature flag:** `settings.allow_employee_receive` (default = `false`) — ถ้าเจ้าของไว้ใจและต้องการให้พนักงานรับของเข้าได้ ค่อยเปิดเป็น `true` ภายหลัง (การเปิด/ปิด cash session และการดูต้นทุน/กำไร ยังเป็น owner-only เสมอ ไม่มี flag)

## 5. ขอบเขต MVP (MVP Scope)

โมดูลหลัก 11 ตัว:

1. **Login / Auth** — username + PIN
2. **Owner Dashboard** — การ์ดแรก "รายได้วันนี้"
3. **POS หน้าขาย** — ค้นหาด้วยชื่อสินค้าภาษาไทย (ไม่มีบาร์โค้ด)
4. **จัดการสินค้า** — เพิ่ม/แก้ ต้นทุน ราคา สต็อก min_stock
5. **ประวัติการขายรายวัน** — ดูตามวัน
6. **Cash Session** — เปิดเงินทอน / ปิดยอดสิ้นวัน / เทียบเงิน
7. **รายงาน** — ยอดขาย กำไร ของขายดี
8. **จัดการพนักงาน** — เพิ่ม/ปิดสิทธิ์
9. **Activity Logs** — ใคร-ทำอะไร-เมื่อไหร่
10. **LINE Notification** — แจ้งเตือนการขายผ่าน LINE Messaging API
11. **Low Stock Alert** — เตือนเมื่อ stock_qty ≤ min_stock

ระบบเป็น **online-first** เท่านั้น

## 6. นอกขอบเขต MVP (Out of Scope)

- ❌ บาร์โค้ดทุกรูปแบบ (สแกน, กรอก, สร้าง, พิมพ์ฉลาก, สแกนกล้อง, ค้นด้วยบาร์โค้ด)
- ❌ Offline mode / IndexedDB / Service Worker
- ❌ ระบบบัญชีเต็มรูป, e-Tax, ใบกำกับภาษี
- ❌ หลายสาขา (เตรียม `store_id` ไว้ในฐานข้อมูลแล้ว แต่ UI ยังเป็นร้านเดียว)
- ❌ ขายเชื่อ/ลูกหนี้, โปรโมชัน/แต้มสมาชิก
- ❌ AI พยากรณ์ยอดขาย, แอป native

> เหตุผลตัดบาร์โค้ด: ลูกค้ายืนยันไม่ต้องการ ร้านเล็กสินค้าหลากหลาย หลายชิ้นไม่มีบาร์โค้ด (บุหรี่แยกซอง เหล้าแบ่งขาย) การพิมพ์ฉลากเองเพิ่มภาระ → ค้นด้วยชื่อ + ปุ่มขายดีเร็วพอและง่ายกว่า
> เหตุผลตัด offline: เพิ่มความซับซ้อนสูง (sync conflict) ขัดหลัก "ไม่ซับซ้อน" → เลื่อนไป Future

## 7. User Workflows

### 7.1 Owner Workflow
1. เช้า: ล็อกอิน → **เปิด cash session** (กรอกเงินทอนตั้งต้น) → ดู Dashboard (รายได้เมื่อวาน, ของใกล้หมด)
2. ระหว่างวัน: รับ LINE แจ้งเตือนทุกการขาย ดู Dashboard ผ่านมือถือ, รับของเข้า, แก้ต้นทุน/ราคาสินค้า
3. เย็น: **ปิด cash session** นับเงินจริงเทียบส่วนต่าง
4. สิ้นเดือน: ดูรายงานกำไร, ของขายดี, ของจะหมด, activity log

### 7.2 Employee Workflow
1. เข้ากะ: ล็อกอิน (เจ้าของเป็นผู้เปิด cash session ให้)
2. ระหว่างกะ: ขายสินค้า / ค้นหาสินค้าด้วยชื่อ / เช็คสต็อก (จำนวน)
3. ออกกะ: ล็อกเอาท์ (การปิด cash session + นับเงินเป็นหน้าที่เจ้าของ)

> รับของเข้าเป็นสิทธิ์เจ้าของโดยค่าเริ่มต้น พนักงานทำได้เฉพาะเมื่อเปิด flag `allow_employee_receive`

### 7.3 POS Sales Workflow (ไม่มีบาร์โค้ด)
1. พนักงานพิมพ์ **ชื่อสินค้าภาษาไทย** ในช่องค้นหา (หรือแตะปุ่มสินค้าขายดี)
2. ระบบกรองรายการสด (trigram partial match เช่นพิมพ์ "มาม่า" เจอ "มาม่าต้มยำกุ้ง")
3. แตะสินค้า → เพิ่มเข้าตะกร้า → ปรับจำนวนได้
4. กด "รับเงิน" → เลือกวิธีจ่าย (เงินสด/พร้อมเพย์) → กรอกเงินรับ → ระบบคำนวณเงินทอน
5. ยืนยัน → บันทึกบิล (status=completed) → **ตัดสต็อกอัตโนมัติ** (สร้าง inventory_movement) → **ส่ง LINE แจ้งเตือน (async)**
6. ยกเลิกบิลทำได้เฉพาะ **void พร้อมเหตุผล** (ห้ามลบ) — พนักงาน void ได้เฉพาะบิลของตัวเองในกะ, เจ้าของ void ได้ทุกบิล

### 7.4 Daily Sales History Workflow
1. เจ้าของเลือกเมนู "ประวัติการขาย" → เลือกวันที่
2. ระบบแสดงรายการบิลของวันนั้น: **วันที่ · เวลา · พนักงาน · สินค้า · จำนวน · ยอดรวม · วิธีจ่าย · สถานะ(completed/void)**
3. แตะบิลเพื่อดูรายละเอียด sale_items
4. แสดงยอดรวมของวัน (เฉพาะ completed)

### 7.5 Cash Session Workflow
1. เปิดกะ (**เจ้าของเท่านั้น**): กรอก `opening_cash` (เงินทอนตั้งต้น) — 1 ร้านมีกะเปิดได้ครั้งละ 1 กะ
2. ระบบสะสมยอดขายเงินสดระหว่างกะ → `expected_cash = opening_cash + ยอดขายเงินสด`
3. ปิดกะ (**เจ้าของเท่านั้น**): นับเงินจริง กรอก `actual_cash`
4. ระบบคำนวณ `difference = actual_cash - expected_cash` → ถ้าติดลบ/เกิน เด้งเตือนเจ้าของ
5. ปิดแล้วแก้ไม่ได้ (immutable) — เก็บเป็นหลักฐาน

### 7.6 LINE Notification Workflow
1. เมื่อบิล status=completed → trigger ส่งแจ้งเตือน
2. ส่งแบบ **async** (ไม่บล็อกหน้าขาย) ผ่าน LINE Messaging API (push message หา owner user id)
3. บันทึกผลใน `notification_logs` (sent/failed + error)
4. ถ้า fail → ทำเครื่องหมายไว้ retry ได้ (ไม่ทำให้การขายล้มเหลว)

**ตัวอย่างข้อความ:**
```
มีการขายสินค้าใหม่

พนักงาน: สมชาย
ยอดขาย: 245 บาท
เวลา: 14:32
สินค้า:
- โค้ก x2
- มาม่า x5
```

## 8. Dashboard Requirements

การ์ดเรียงตามลำดับ:
1. **"รายได้วันนี้"** (ตัวเลขใหญ่สุด เด่นสุด) — ยอดขายรวม completed ของวันนี้
2. กำไรวันนี้ (เจ้าของเท่านั้น)
3. เงินสดในลิ้นชัก (จาก cash session ที่เปิดอยู่)
4. ของใกล้หมด (จำนวนรายการที่ stock_qty ≤ min_stock)

ด้านล่าง: กราฟยอดขาย 7 วัน, รายการสินค้าขายดี, รายการของใกล้หมด

## 9. กฎสิทธิ์ (Permission Rules)

| การกระทำ | Owner | Employee |
|---|:---:|:---:|
| ขายสินค้า | ✅ | ✅ |
| ค้นหาสินค้าด้วยชื่อ | ✅ | ✅ |
| เช็คสต็อก (จำนวน) | ✅ | ✅ |
| รับของเข้า | ✅ | ❌ default (เปิดได้ด้วย flag `allow_employee_receive`) |
| แก้ต้นทุนสินค้า | ✅ | ❌ |
| เปิด cash session | ✅ | ❌ |
| ปิด cash session | ✅ | ❌ |
| ดูต้นทุน/กำไร | ✅ | ❌ |
| ดูรายงาน sensitive | ✅ | ❌ |
| ดูประวัติการขายทุกวัน | ✅ | ❌ (เห็นเฉพาะกะตัวเอง) |
| void บิล | ✅ ทุกบิล | ❌ |
| ลบบิล | ❌ (ห้ามทุกคน) | ❌ |
| จัดการสินค้า | ✅ | ❌ |
| จัดการพนักงาน | ✅ | ❌ |
| ตั้งค่าระบบ/LINE | ✅ | ❌ |
| ดู/ลบ activity log | ดูได้ (ลบไม่ได้) | ❌ |

**Feature flag เดียวใน MVP:** `settings.allow_employee_receive` (default `false`) — คุมเฉพาะสิทธิ์ "รับของเข้า" ของพนักงาน สิทธิ์อื่น (cash session, ต้นทุน/กำไร, void) เป็น owner-only เด็ดขาด ไม่มี flag

บังคับสิทธิ์ทั้งระดับ **app** (ซ่อน UI) และระดับ **DB** (RLS + เช็ค role ใน RPC functions)

## 10. กฎ Activity Log

ต้องบันทึกทุกการกระทำสำคัญ: login/logout, สร้าง/แก้/ปิดใช้สินค้า, แก้ราคา/ต้นทุน, รับของเข้า, ปรับสต็อก, void บิล, เปิด/ปิด cash session, เพิ่ม/แก้/ปิดพนักงาน, แก้ตั้งค่า
- เก็บ: user, action, entity, entity_id, detail (JSON ก่อน/หลัง), เวลา
- **ห้ามแก้ ห้ามลบ** (append-only)

## 12. Future Roadmap

- **V2:** ขายเชื่อ/ลูกหนี้, ตรวจนับสต็อก (audit นับจริงเทียบระบบ), แจ้งเตือนของใกล้หมดอายุ, รายงานกำไรเชิงลึก, บาร์โค้ด (ถ้าลูกค้าเปลี่ยนใจ)

## 13. ความเสี่ยง (Risks)

| ความเสี่ยง | การรับมือ |
|---|---|
| เน็ตร้านล่ม → ขายไม่ได้ (online-first) | ยอมรับใน MVP; offline เป็น Future; แนะนำเน็ตสำรอง 4G |
| พนักงานเลี่ยงระบบ ("จดมือเร็วกว่า") | UX ต้องเร็วกว่าจดมือ + ปุ่มขายดี + ค้นชื่อไว |
| เก็บเงินเองไม่ลงระบบ | cash session เทียบเงิน + กล้องวงจรปิด (กระบวนการนอกระบบ) |
| ต้นทุนไม่อัปเดต → กำไรเพี้ยน | เก็บ unit_cost ณ เวลาขายใน sale_items |
| LINE ส่งไม่เข้า | async + notification_logs + retry |
| Over-engineering | ยึด scope MVP เคร่งครัด |

## 14. เกณฑ์ความสำเร็จ (Success Criteria)

- พนักงานใหม่ขายบิลแรกได้ภายใน 5 นาทีโดยไม่ต้องสอนมาก
- ขาย 1 รายการเสร็จใน < 10 วินาที (ค้นชื่อ → ตะกร้า → รับเงิน)
- เจ้าของเห็น "รายได้วันนี้" ทันทีที่เปิด Dashboard
- ทุกการขายส่ง LINE ถึงเจ้าของภายในไม่กี่วินาที
- ส่วนต่างเงินสดสิ้นวันตรวจสอบได้ทุกกะ
- ไม่มีบิลถูกลบออกจากระบบ (มีแต่ void ที่ตรวจสอบได้)
